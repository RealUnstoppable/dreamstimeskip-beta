## 2025-02-14 - Fix Privilege Escalation in Firestore Rules
**Vulnerability:** Insecure Direct Object Reference (IDOR) and Privilege Escalation in Firestore `users` collection rules. Authenticated users were allowed to write to any field in their own profile document, including sensitive fields like `isAdmin` and `membershipLevel`.
**Learning:** Default "allow write" rules without field-level restrictions in Firebase allow users to maliciously elevate their privileges or alter billing-related fields directly via the Client SDK, bypassing the application frontend entirely.
**Prevention:** Always separate `write` into `create`, `update`, and `delete`. For `create`, strictly verify default values using `.get('field', default)` or prohibit restricted keys using `!request.resource.data.keys().hasAny([...])`. For `update`, use `request.resource.data.diff(resource.data).affectedKeys().hasAny([...])` to prevent changes to sensitive role or subscription fields. Relying solely on `request.auth.uid == userId` is insufficient for collections containing role-based data.
